### **1. 版本号格式**
使用 `V<主版本号>.<次版本号>.<修订号>` 格式（例如：`V2.96.0`）。

### **2. 递增逻辑**

#### **主版本号 (Major) - 第一位数字**
*   **触发条件**：项目架构重大调整、底层框架更换、或出现不兼容的功能更新。
*   **示例**：从 Python 2 迁移到 Python 3；UI 框架从 Tkinter 彻底更换为 PySide/PyQt。
*   **变化**：`V2.96.5` -> `V3.0.0`

#### **次版本号 (Minor) - 第二位数字**
*   **触发条件**：新增了独立的功能模块、新的脚本工具、或重要的逻辑增强。
*   **示例**：新增了 `generate_formatted_columns.py` 脚本；增加了对新电表协议的支持。
*   **变化**：`V2.96.5` -> `V2.97.0`

#### **修订号 (Patch) - 第三位数字**
*   **触发条件**：修复 Bug、优化现有逻辑的代码性能、微调界面布局、或修改文案。
*   **示例**：修复了 Excel 写入覆盖的问题；修改了某个按钮的颜色；更新了注释。
*   **变化**：`V2.96.5` -> `V2.96.6`

---

### **3. 更新流程规范**

为了配合 [project_rules.md](file:///e:/code/需求文档出厂参数项提取脚本/.trae/rules/project_rules.md) 的要求，每次更新应遵循以下流程：

1.  **代码变更**：完成代码修改并自测通过。
2.  **确定版本**：
    *   如果是 **Bug 修复/小优化**：修订号 +1。
    *   如果是 **新功能/脚本**：次版本号 +1，修订号归零。
    *   如果是 **重大重构**：主版本号 +1，次版本号和修订号归零。
3.  **记录日志**：在 [updates_notes.txt](file:///e:/code/需求文档出厂参数项提取脚本/updates_notes.txt) 的**最顶部**添加新块：
    ```text
    V2.97.0 2025-12-26
    - 新增：[功能名称] 详细描述。
    - 优化：[优化点] 详细描述。
    - 修复：[Bug描述] 解决了什么问题。
    ```
4.  **同步代码**：如果代码中有硬编码的版本号（如 [main_window.py](file:///e:/code/需求文档出厂参数项提取脚本/src/ui/main_window.py) 中的读取逻辑），确保它们能正确读取到最新版本。
